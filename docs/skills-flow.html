Skill Details & Flows

📍

/where-am-i

"project status"
Quick scannable briefing — what's done, in progress, next steps, any drift.
  1. READ docs/implementation-phases.md — parse [x] / [ ] checkboxes
  2. READ docs/README.md — project overview
  3. Scan codebase: ls backend/app/api/v1/, services, models, schemas, repos, tests
  4. Compare scan results vs plan checkboxes → detect drift
  5. Output: Done phases, current phase, next 3-5 tasks, drift warnings
READS plan
⚡

/execute-next

"keep going"
Primary orchestrator. Picks next task, runs full TDD cycle, updates plan.
  1. READ docs/implementation-phases.md — find earliest [ ] task
  2. Verify all prior phases are complete (no skipping)
  3. If no SPEC.md → calls /create-spec to generate one
  4. Read SPEC.md + business-rules + api-routes + db-schema
  5. Write failing tests FIRST (Red phase)
  6. Implement minimum code to pass (Green phase)
  7. Refactor while keeping tests green
  8. WRITE back to plan: change [ ] → [x]
  9. Run /code-coverage for consistency check
READS plan WRITES plan CALLS /create-spec CALLS /code-coverage
📝

/create-spec

"spec out auth"
Creates SPEC.md for a feature module. Retroactive (document code) or Proactive (design guide).
  1. Detect project structure (API dir, services, schemas, tests)
  2. Identify module name + feature doc + source files
  3. Mode detect: Retroactive (code exists) vs Proactive (doesn't)
  4. Read feature doc + API routes + business rules + DB schema
  5. If retroactive: also read schemas, services, routes, repos, tests
  6. Analyze: routes, access control, schemas, edge cases
  7. Write docs/specs/{module}.SPEC.md with template
  8. Verify: file paths exist, response codes match, schema names match
READS feature docs
🧪

/tdd-pipeline

"write tests first"
Complete TDD pipeline: spec → test cases → coverage analysis → pytest code → report.
  1. Detect project structure (pytest config, test dirs, import root)
  2. plan: Extract test cases from SPEC.md → TEST-CASES.md
  3. analyze: Map spec requirements to test cases → find gaps
  4. generate: Write runnable pytest files (unit, service, API)
  5. Verify RED: pytest -v — tests must fail
  6. report: Run tests, produce pass/fail summary
READS SPEC.md
🔍

/code-coverage

"check the plan"
Audits plan vs actual codebase. Finds STUB, MISSING, DRIFT, UNPLANNED, OUT OF ORDER.
  1. READ docs/implementation-phases.md — build done/incomplete lists
  2. Scan codebase: file existence, line counts, stub detection
  3. Check for spec alignment (SPEC.md vs code)
  4. Check dependency violations (Phase N+1 before Phase N)
  5. Check for unplanned work (files outside the plan)
  6. Classify: MISSING, STUB, DRIFT, UNPLANNED, OUT OF ORDER, PLAN GAP
  7. Output: Progress table + deviations + risk + recommendations
READS plan
✅

/test-auditor

"audit my tests"
Audits pytest tests against spec AND implementation. Finds incorrect, missing, redundant tests.
  1. Discover project layout (tests, specs, implementation)
  2. Load 3 sources: spec docs, implementation code, test files
  3. Audit A: Tests vs Spec → find INCORRECT tests
  4. Audit B: Spec → Tests → find MISSING coverage
  5. Audit C: Implementation → Tests → find untested code paths
  6. Audit D: Redundancy → find duplicate/trivial/dead tests
  7. Classify by severity: Critical / High / Medium / Low
  8. Output: Summary table + per-finding detail + recommendations
READS specs + code + tests
🛡

/security-audit

"check security"
Full security scan: secrets, auth, API, database, infra, dependencies.
  1. READ docs/implementation-phases.md — what's been built
  2. Read DB schema, business rules, API routes, .env.example
  3. Scan: hardcoded secrets, JWT config, password hashing
  4. Scan: CORS, input validation, raw SQL, rate limiting
  5. Scan: Docker config, port exposure, DEBUG mode
  6. Scan: dependency CVEs, lockfiles
  7. Output: docs/audit-report.md with findings + code fixes
  8. Suggest additions to implementation-phases.md
READS plan SUGGESTS plan changes
📚

/update-notes

"update notes"
Writes beginner-friendly learning notes into notes/ based on recently built features.
  1. READ docs/implementation-phases.md — which tasks are [x] done
  2. Scan recently modified Python/TypeScript files
  3. Read existing notes (avoid duplication)
  4. Map completed work to topic files (e.g., auth → auth-and-security.md)
  5. Write/update notes with: concepts, code examples, commands, gotchas
  6. Update notes/README.md index
READS plan
🔨

/skill-creator

"create a skill"
Meta-skill: creates, tests, benchmarks, and optimizes other skills. Does not directly use implementation-phases.md.
  1. Capture intent: what should the skill do, when should it trigger
  2. Interview for edge cases, input/output formats, success criteria
  3. Write SKILL.md with frontmatter + instructions
  4. Create 2-3 test prompts → spawn with-skill + baseline subagents
  5. Grade assertions + aggregate benchmark + launch eval viewer
  6. Read user feedback → improve skill → repeat until satisfied
  7. Optimize description for triggering accuracy (run_loop.py)
CREATES other skills

Typical Development Workflow

  1. 1
    /where-am-i — "Where did I leave off?" Get a status briefing.
  2. 2

  3. ↳

    /tdd-pipeline — Pattern used internally: Red → Green → Refactor.

  9. ⚙
    /skill-creator — As needed: create or improve any skill in the ecosystem.
